A knowledge base and wiki that publishes like a blog.

Index Cards is a self-hosted knowledge base where every page is both a wiki entry and a blog post. Write it, link it to whatever else it's about, and publish it when it's ready — no separate systems, no import steps between them.

Spaced Repetition

Also known as: distributed practice

A technique for reviewing material at increasing intervals, timed to land just before you'd otherwise forget it. Closely related to the forgetting curve and formalized in systems like the Leitner system.

Most flashcard software implements some version of this — see also active recall for the retrieval side of it.

Day to day

From draft to someone's inbox

Write it as a draft

The editor is a real WYSIWYG (ProseMirror) that reads and writes plain Markdown underneath. Drafts are only visible to writers, never to the public.

Link it as you go

Typing [[ opens an inline search over existing titles. Link to an entry that doesn't exist yet and it renders as a distinct "missing" link — a visible prompt to write it later, the same way a classic wiki works.

Publish when it's ready

Flip it out of draft and it appears in the index, the RSS/Atom/JSON feeds, and — if the entry is new that week — the digest queue. Publishing can also announce to Slack or fire an outgoing webhook, if configured.

Subscribers get a weekly note

Anyone who subscribed gets one email a week listing what's new and what changed. No mailing list service required — it's just SMTP, with Mailchimp sync as an optional add-on.

Edit it later, history stays

Every save that changes content is a revision you can review and restore. Concurrent editors get a soft lock so two people don't silently overwrite each other's work on the same page.

Find it again instantly

Full-text search runs on SQLite's FTS5 engine — no external search service, no indexing pipeline to babysit. Ranked results and highlighted snippets are built in.

Wiki mechanics

Aliases
An entry can have alternate titles that redirect to it — cover alternative terms and rename freely without breaking old links or bookmarks.
Backlinks
Every internal link is tracked in both directions and rebuilt automatically on save. Never maintain your references by hand.
Hierarchy
Entries can optionally have a parent, for grouping subpages under a topic.
Missing-link prompts
Links to entries that don't exist yet render distinctly, so gaps in the wiki are visible instead of silently broken.

Blog mechanics

Drafts
Write and revise privately; nothing is public until it's explicitly published.
Feeds
RSS, Atom, and JSON feed formats generated automatically from published entries.
Weekly digest
One email per week to subscribers, summarizing new and (optionally) edited entries, on a configurable day.
Revision history
Every content-changing save is kept as a restorable snapshot, with an optional changelog note per edit.

Design

A few deliberate choices

One object, not two

An entry is simultaneously a wiki page and a blog post — draft, published, and everything in between are just states of the same record, not separate content types.

No lock-in

Content is stored as Markdown and exportable as Markdown or JSON at any time. Nothing about the format depends on your app staying online.

Small footprint

One SQLite file, one Python process, no external search index, message queue, or cache layer required to run it.

Self-hosted by default

Built to run on a single low-cost server you control, with a setup script that gets a production instance running in minutes.

Questions

FAQ

Is this a wiki or a blog?

Both, on purpose. Entries behave like wiki pages (aliases, backlinks, optional hierarchy) and like blog posts (drafts, publish dates, feeds, a digest) at the same time — there's no separate "post" content type to keep in sync.

Can more than one person write on the same site?

Yes. There are four roles (admin, editor, author, viewer), invite-based or open signup, and a soft edit lock so two people don't silently overwrite the same entry.

What happens to my content if I stop using it?

Everything is exportable as Markdown or JSON from the admin dashboard at any time. The database itself is a single SQLite file, so there's also nothing preventing direct access to the raw data.

Does it need a mail service?

It needs SMTP credentials to send magic-link logins, invites, and the weekly digest. If none are configured, emails print to the console instead — convenient for local development, not for production.

What does hosting it cost?

Whatever a small Ubuntu/Debian VM costs — there's no database service, search service, or queue to pay for separately. SQLite and the app run on the same box. It's possible to host this app for a couple bucks per month.

Can I change how it looks?

Yes — pick from several built-in color themes, or go further with custom CSS and injected head/footer HTML from the settings page.
